> 版本：v2.0 (评审整合版)
> 日期：2026-07-08
> 状态：设计完成（含多视角评审修正），待 MVP spike 验证

## 一、项目概述

**项目名：** Novel Eval
**仓库：** `reed-soul/novel-eval`

**定位：** 个人使用的 AI 小说评估工具。上传小说文件 → 五维结构化评估 → 交互式 HTML 报告。

**核心流程：**
```
上传文件(txt/epub/pdf/docx)
  → 文档解析(统一接口层)
  → 结构化分章(解析优先，LLM fallback)
  → Map阶段: 逐章独立评估(Anthropic API 直调，Claude Code fallback)
  → Reduce阶段: 基于全部章节摘要做全局分析(单次 API 调用)
  → 生成结构化评分数据(JSON + checkpoint)
  → 渲染交互式 HTML 报告(雷达图/情绪曲线/事件时间线)
```

**技术栈：**
- 运行环境：Node.js，跑在本地 Mac mini
- 评估引擎：Anthropic API（主力，`AnthropicAPIAdapter` 直调），Claude Code（fallback，`ClaudeCodeAdapter`）
- 文档解析：epub / pdf-parse / mammoth / chardet + iconv-lite
- 输出：JSON 评分数据 + 交互式 HTML 报告（ECharts 内嵌 min.js，离线可用）+ `report.data.json` 数据文件
- 评估规则：项目内 YAML 配置文件，支持多种使用模式 preset
- 语言：TypeScript

## 二、五维评估模型

| 维度 | 子项 | 评分范围 | 权重 |
|------|------|---------|------|
| **故事架构** | 情节完整性、叙事结构合理性、起承转合、节奏把控、悬念设置 | 0-100 | 25% |
| **人物塑造** | 角色立体感、人物弧光完整性、动机合理性、角色关系丰富度 | 0-100 | 25% |
| **文笔质量** | 语言表现力、对话自然度、修辞运用、描写细腻度、风格统一性 | 0-100 | 15% |
| **情感共鸣** | 情感张力、读者代入感、情绪节奏、共鸣深度 | 0-100 | 20% |
| **市场潜力** | 类型定位清晰度、目标受众匹配、创新性、同类作品差异化 | 0-100 | 15% |

> **五维评分全部在 Reduce 阶段产出。** Map 阶段不产出五维评分（市场潜力无法按章评估，其他维度按章评也不够准确）。

### 使用模式 Preset

YAML 中可定义多个评估 profile，适应不同使用场景：

```yaml
# config/profiles.yml
profiles:
  default:
    weights:
      storyStructure: 0.25
      characterization: 0.25
      writingQuality: 0.15
      emotionalResonance: 0.20
      marketPotential: 0.15

  revision:   # 改稿模式：市场潜力权重降低，文笔和结构权重提升
    weights:
      storyStructure: 0.30
      characterization: 0.25
      writingQuality: 0.25
      emotionalResonance: 0.15
      marketPotential: 0.05

  submission: # 投稿模式：市场潜力权重提升
    weights:
      storyStructure: 0.20
      characterization: 0.20
      writingQuality: 0.15
      emotionalResonance: 0.15
      marketPotential: 0.30
```

### 评估产出结构（JSON）

```json
    "emotionalResonance": { "score": 0, "subscores": {}, "analysis": "", "quotes": [] },
    "marketPotential": { "score": 0, "subscores": {}, "analysis": "", "quotes": [] }

    }
  ],
  "emotionalCurve": [
    { "chapterId": "ch001", "tension": 0 }

    { "text": "", "chapterId": "ch001", "offset": 0, "context": "" }
  ],
  "suggestions": [
    { "dimension": "", "type": "", "content": "", "relatedChapters": [], "quote": "" }

  "task": {
    "id": "",
    "error": null,

    "sourceWordCount": 0,

    "completedAt": ""
  }

```

**核心原则：**
- 每个评分必须附带 `quotes`（原文引用，含定位信息）和 `analysis`（分析依据）
- 不做空泛评价，不写"值得点赞"等 AI 味评语
- 权重可在 YAML 配置中调整，支持按 profile 切换
- `quotes` 包含 `chapterId` + `offset`，支撑"展开上下文"和"跳转源文件"
- `emotionalCurve` 只保留 `tension`（valence 删掉，Map 不产出）
- Chapter `id` 改为 string 类型（支持 "vol1-ch003" 等），增加 `kind` 字段

### 小说特有检查项（在 suggestions 中体现，不单独加维度）

以下检查项融入 Reduce 阶段的 suggestions 输出：

- **世界观/设定一致性**：跨章节检测设定矛盾（如角色能力变化、地理描述不一致）
- **视角/叙事一致性**：检测视角跳跃或叙述者身份混乱
- **信息密度/注水检测**：识别低信息密度段落（大段环境描写无情节推进、重复表达）
- **章末钩子/追读力**：评估每章结尾是否设置了悬念或钩子，追读力强弱

## 三、评估流程（Map-Reduce 架构）

### 阶段一：解析 + 分章

**策略：结构解析优先，LLM fallback。**

```
文件上传 → 文档解析(统一接口) → 提取纯文本 → 结构化分章
  输出：[{ id: "ch001", title, content, startIndex, endIndex, kind }]
```

分章按优先级逐级尝试，上一级成功即跳过后续：

| 优先级 | 策略 | 适用格式 | 实现方式 |
|--------|------|---------|---------|
| 1 | 内置章节结构 API | EPUB/DOCX | 直接读取文档章节元数据，零成本 |
| 2 | 正则匹配 | TXT/网文 | 匹配"第X章/回/节" + 空行分隔模式 |
| 3 | 启发式分页 | PDF | 按页/段落启发式切分 |
| 4 | LLM 辅助 | 所有格式（fallback） | 仅对以上方法失败的区间调用 LLM，不是全书 |

> MVP 阶段仅实现正则分章（TXT），LLM 分章放 v2。

### 阶段二：Map — 逐章评估（独立，无前章依赖）

```
章节列表 → 并发调用 Anthropic API（默认 5 并发）
  每章输入：章节正文 + 评估规则 YAML
  每章输出：{
    summary: 章节摘要,
    emotionalTension: 本章情绪张力值(0-100),
    keyEvents: 关键事件列表,
    characters: 本章出场角色
  }
```

**关键设计决策：**
- **Map 只产出 4 项**：摘要、情绪张力、关键事件、出场角色。五维评分全部在 Reduce 阶段计算。
- **并发 Map 取消前章摘要依赖**：每章独立评估，跨章衔接全部交给 Reduce。并发时不传递前章摘要（因为并行时前章可能还没完成）。
- 并发控制：默认 5 并发，可配置。带任务级和单章级预算控制。

**Checkpoint 机制：**
Map 阶段每完成一章即将结果写入 partial JSON。支持中途查看已完成部分，也支持中断恢复。

```typescript
interface EvaluationCheckpoint {
  phase: 'split' | 'map' | 'reduce'
  completedChapterIds: string[]
  partialResultPath: string
  sessionId?: string
}
```

### 阶段三：Reduce — 全局分析

**默认方式：单次 Anthropic API 调用**

```
所有章节评估结果 → 单次 API 调用（传入全部摘要，~100K token < 200K 上下文）
  输入：全部章节摘要 + 人物出场记录 + 情绪张力序列 + 评估规则
  输出：{
    dimensions: 五维全局评分（含 subscore/analysis/quotes）,
    characters: 统一角色列表（归一化处理）,
    emotionalCurve: 整体情绪节奏曲线,
    suggestions: 跨章节改进建议（附原文引用和定位信息）
  }
```

> Claude Code 的 `mcp serve` 是暴露给外部 MCP Client 用的，不做会话编排。Reduce 阶段保留 `stream-json` stdin/stdout 作为 Claude Code fallback 的会话方式，但默认走 Anthropic API 直调。

**树形 Reduce（章节数 > 200 时）：**
防止摘要总量超上下文窗口。先每 ~20 章 Local Reduce（生成弧摘要），再 Global Reduce 汇总所有弧摘要。

```
[ch1-ch20] → Local Reduce → 弧摘要1
[ch21-ch40] → Local Reduce → 弧摘要2
...
[ch181-ch200] → Local Reduce → 弧摘要10
    ↓
[弧摘要1..10] → Global Reduce → 最终全局分析
```

**人物归一化：**
Reduce 阶段用 LLM 合并同名/昵称/代称（如 "张三"/"张 三"/"三哥"），输出统一角色列表。

### 评分聚合公式

```typescript
// 1. 章级子分 → 维度分（仅限章级有意义的维度：故事架构、人物塑造、文笔质量、情感共鸣）
dimensionScore = Σ(chapterSubscore × chapterWordWeight)

// 2. 维度分 → 总分
overall.totalScore = Σ(dimensionScore × yamlWeight)

// 3. 总分 → 等级
grade = 查表(yaml thresholds)
```

```yaml
# config/default.yml
gradeThresholds:
  S: 90
  A: 80
  B: 70
  C: 60
  D: 0
```

### 错误处理与重试

- **单章评估失败** → 重试 1 次（JSON Schema 校验失败 → 自动重试，最多 2 次）→ 仍失败则标记跳过，最终报告标注缺失章节
- **Reduce 阶段中断** → 可基于已有 checkpoint 恢复，不需要重新 Map
- **LLM 输出校验**：强制 JSON Schema 校验，失败自动重试（最多 2 次），超过则报错

### 成本与超时控制

```typescript
interface CostConfig {
  // 任务级预算
  taskMaxBudgetUsd: number;       // 默认 50
  // 单章预算
  perChapterMaxBudgetUsd: number; // 默认 0.5
  // 动态超时：baseMs + wordCount * factor
  timeout: {
    baseMs: number;               // 默认 30000
    perThousandWordsMs: number;  // 默认 200
  };
  // watchdog：流空闲超时
  streamIdleTimeoutMs: number;   // 默认 90000 (90s)
}
```

- watchdog：流空闲 90s 无新事件 → kill 当前调用
- 单次调用超时：`30000ms + wordCount * 0.2ms/千字`
- 任务级总预算硬限制：$50（可配置）

## 四、统一抽象层

### 核心接口

```typescript
interface AIAgentEvent {
  type: 'assistant' | 'tool_use' | 'result' | 'error';
  content: string;
  data?: any;
  usage?: { inputTokens: number; outputTokens: number; costUsd: number };
}

interface AIAgentAdapter {
  readonly name: string;

  // 单次调用
  run(prompt: string, options: RunOptions): AsyncIterable<AIAgentEvent>;

  // 会话模式（Claude Code fallback via stream-json stdin/stdout）
  createSession(options?: SessionOptions): AIAgentSession;

  // 查询可用性
  isAvailable(): Promise<boolean>;
}

interface AIAgentSession {
  send(prompt: string): AsyncIterable<AIAgentEvent>;
  close(): void;
}

interface RunOptions {
  cwd?: string;
  systemPrompt?: string;
  allowedTools?: string[];
  disallowedTools?: string[];
  maxBudgetUsd?: number;
  model?: string;
  timeoutMs?: number;
  outputSchema?: object;
}

interface SessionOptions {
  cwd?: string;
  systemPrompt?: string;
  model?: string;
}
```

### Adapter 实现

| Adapter | `run()` 底层 | `createSession()` 底层 | 优先级 |
|---------|-------------|----------------------|--------|
| **`AnthropicAPIAdapter`** | Anthropic SDK（`messages.create`，支持 prompt caching） | — | **默认** |
| `ClaudeCodeAdapter` | `claude -p --output-format stream-json` | stream-json stdin/stdout | fallback |

> **AnthropicAPIAdapter 为默认实现**。理由：无冷启动开销、prompt caching 省 ~$2.7/本、API 更稳定、无已知 stream-json 并发挂起 bug。ClaudeCodeAdapter 保留为 fallback（本地无 API key 但有 Claude Code 时可用）。

### 引擎配置（engines.yml）

```yaml
default: anthropic-api
fallback: claude-code

anthropic-api:
  model: claude-sonnet-4-20250514
  maxBudgetUsd: 50
  perChapterMaxBudgetUsd: 0.5
  promptCaching: true

claude-code:
  model: claude-sonnet-4-20250514
  maxBudgetUsd: 5
  allowedTools: [Read, Glob, Grep]
```

### 关键设计决策

- 所有 adapter 实现同一接口，上层代码只依赖 `AIAgentAdapter`
- `run()` 返回 `AsyncIterable`，支持流式处理（进度条、实时更新）
- 评估场景只开放只读工具（Read、Glob、Grep），禁止文件写入
- 超时和预算双重兜底，防止失控
- LLM 输出强制 JSON Schema 校验 + 自动重试（最多 2 次）

## 五、文档解析层

### 统一接口

```typescript
interface ParserAdapter {
  readonly supportedExtensions: string[];
  parse(filePath: string): Promise<ParsedDocument>;
}

interface ParsedDocument {
  title?: string;
  author?: string;
  text: string;       // 纯文本全文
  metadata?: Record<string, any>;
}
```

### 各格式实现

| 格式 | 库 | 说明 |
|------|-----|------|
| TXT | `chardet` + `iconv-lite` | 自动检测编码（GBK/UTF-8等），转为 UTF-8 |
| EPUB | `epub` / `epub2md` | 提取章节 HTML → 剥离标签 → 纯文本 |
| PDF | `pdf-parse` | 基于 pdf.js，纯 JS 无原生依赖 |
| DOCX | `mammoth` | DOCX → 纯文本/Markdown |

> MVP 仅实现 TXT 解析。EPUB/PDF/DOCX 放 v2。

### 自动注册

`registry.ts` 根据文件扩展名自动选择解析器，新增格式只需实现 `ParserAdapter` 接口并注册。

首次启动检测 Claude API key / Claude Code 可用性：
```
$ novel-eval evaluate ./book.txt

✗ No Anthropic API key found (ANTHROPIC_API_KEY)
  → Set environment variable: export ANTHROPIC_API_KEY=sk-ant-...
  → Or install Claude Code: claude will be used as fallback
```

### 评估前确认屏

CLI 发起评估前显示预估信息，用户确认后开始：
```
$ novel-eval evaluate ./book.txt

📖 Estimated chapters: ~85 (regex split)
⏱ Estimated duration: ~18 min
💰 Estimated cost: $5.2 - $10.5 (Sonnet, with caching)

Proceed? [Y/n]
```

### 评估中实时进度

```
✓ Split: 87 chapters identified
⚡ Mapping [████████████░░░░░░░░] 52/87 chapters
  Ch.52 "第四十二章·雨夜" — done (1,823 tokens, $0.03)
  💰 Cumulative: $2.41 | ⏱ Elapsed: 10m23s | ETA: ~7m
```

### 中途查看

Map 阶段每完成一章即写入 `results/{taskId}.partial.json`，可随时查看已完成部分。


## 七、HTML 报告

### 报告架构

**分层设计，避免 300+ 章撑爆单文件：**
- `report.html`：骨架 + ECharts 图表 + 报告框架
- `report.data.json`：全量评估数据（同目录）
- 章节详情：懒加载，点击展开时从 JSON 读取

**安全：**
- 所有动态内容 HTML escape
- JSON 用 `<script type="application/json">` 内嵌，不拼接字符串
- 内嵌 ECharts min.js（不依赖 CDN，离线可用）

### 可视化优先级

| 优先级 | 可视化 | 说明 |
|--------|--------|------|
| **P0** | 情绪/节奏曲线 | 长篇最缺的宏观视角，面积图 |
| **P0** | 五维雷达图 | 仅全局五维，10 秒看懂长短板 |
| **P1** | 章节事件时间线 | 替代原来的"章节五维折线"（章级五维不再存在） |
| **P2** | 人物关系图谱 | 角色 <15 且关系简单时才有用 |
| ~~砍掉~~ | ~~市场对标看板~~ | 个人本地工具没有真实竞品数据源 |

### 报告页面结构


```typescript
  | 'pending' | 'parsing' | 'splitting'
  | 'mapping' | 'reducing' | 'generating'
```

## 十、成本估算

### 理论成本（100 万字小说）

| 场景 | 模型 | 估算成本 | 备注 |
|------|------|---------|------|
| 纯 Claude API (Sonnet) | claude-sonnet-4 | ~$22/本 (约158元) | 无优化，直接调用 |
| 优化后（推荐） | Haiku Map + Sonnet Reduce + prompt caching | ~$5-10/本 | MVP 使用的方案 |
| Map 逐章 Claude Code | Claude Code CLI | +25% 额外开销 | Claude Code 冷启动 + 进程开销 |

### 成本优化策略

1. **Prompt Caching**：Anthropic API 支持 prompt caching，评估规则 YAML 和 system prompt 缓存后可省 ~$2.7/本
2. **Haiku 混合**：Map 阶段可选用 Haiku（便宜 90%），Reduce 阶段用 Sonnet 保证质量
3. **并发效率**：5 并发 Map 可缩短总时间 ~60%，减少等待成本

### 成本模型（简化）

```
Map 成本 ≈ 章节数 × 平均每章 token × 单价
  - 100万字 ≈ 333章 × ~2000 token/章
Reduce 成本 ≈ 全部摘要 token × 单价
  - 333章摘要 ≈ 66K token
总计 ≈ Map + Reduce + 分章 ≈ $5-22/本（取决于模型选择和优化程度）
```

### 任务预算控制

```yaml
# engines.yml
taskMaxBudgetUsd: 50          # 整次评估任务上限
perChapterMaxBudgetUsd: 0.5  # 单章评估上限
```

## 十一、安全设计

| 风险 | 防护措施 |
|------|---------|
| 文件上传路径穿越 | 文件存 `data/novels/{uuid}.{ext}`，不使用原始文件名 |
| 恶意文件（zip bomb 等） | 扩展名白名单校验 + 文件大小上限 |
| HTML 报告 XSS | 所有动态内容 HTML escape；JSON 用 `<script type="application/json">` 内嵌 |
| 本地 Web 服务暴露 | 绑定 `127.0.0.1`；若需局域网访问，加 token / basic auth |
| 数据泄露 | 数据保留策略：默认 30 天自动清理；README 提醒用户 |
| API 费用失控 | 任务级 + 单章级双重预算上限，超额暂停并写入任务状态 |
