> 状态：设计完成（经 spike 验证：Map 稳定、R2 不截断、成本极低；已修订 quotes 机制 + GLM 化），具备开工条件

## 一、项目概述

**项目名：** Novel Eval
**仓库：** `reed-soul/novel-eval`

**定位：** 个人使用的 AI 小说评估工具。上传小说文件 → 五维结构化评估 → 交互式 HTML 报告。

**核心流程：**
```
上传文件(txt/epub/pdf/docx)
  → 文档解析(统一接口层)
  → 结构化分章(解析优先，LLM fallback)
  → Map阶段: 逐章独立评估(Anthropic API 直调，Claude Code fallback)
  → Reduce阶段: 基于全部章节摘要做全局分析(单次 API 调用)
  → 生成结构化评分数据(JSON + checkpoint)
  → 渲染交互式 HTML 报告(雷达图/情绪曲线/事件时间线)
```

**技术栈：**
- 运行环境：Node.js，跑在本地 Mac mini
- 评估引擎：Anthropic API（主力，`AnthropicAPIAdapter` 直调），Claude Code（fallback，`ClaudeCodeAdapter`）
- 文档解析：epub / pdf-parse / mammoth / chardet + iconv-lite
- 输出：JSON 评分数据 + 交互式 HTML 报告（ECharts 内嵌 min.js，离线可用）+ `report.data.json` 数据文件
- 评估规则：项目内 YAML 配置文件，支持多种使用模式 preset
- 语言：TypeScript

## 二、五维评估模型

| 维度 | 子项 | 评分范围 | 权重 |
|------|------|---------|------|
| **故事架构** | 情节完整性、叙事结构合理性、起承转合、节奏把控、悬念设置 | 0-100 | 25% |
| **人物塑造** | 角色立体感、人物弧光完整性、动机合理性、角色关系丰富度 | 0-100 | 25% |
| **文笔质量** | 语言表现力、对话自然度、修辞运用、描写细腻度、风格统一性 | 0-100 | 15% |
| **情感共鸣** | 情感张力、读者代入感、情绪节奏、共鸣深度 | 0-100 | 20% |
| **市场潜力** | 类型定位清晰度、目标受众匹配、创新性、同类作品差异化 | 0-100 | 15% |

> **五维评分全部在 Reduce 阶段产出。** Map 阶段不产出五维评分（市场潜力无法按章评估，其他维度按章评也不够准确）。

### 使用模式 Preset

YAML 中可定义多个评估 profile，适应不同使用场景：

```yaml
# config/profiles.yml
profiles:
  default:
    weights:
      storyStructure: 0.25
      characterization: 0.25
      writingQuality: 0.15
      emotionalResonance: 0.20
      marketPotential: 0.15

  revision:   # 改稿模式：市场潜力权重降低，文笔和结构权重提升
    weights:
      storyStructure: 0.30
      characterization: 0.25
      writingQuality: 0.25
      emotionalResonance: 0.15
      marketPotential: 0.05

  submission: # 投稿模式：市场潜力权重提升
    weights:
      storyStructure: 0.20
      characterization: 0.20
      writingQuality: 0.15
      emotionalResonance: 0.15
      marketPotential: 0.30
```

**核心原则：**
- 每个评分必须附带**原文证据**（excerpts，含定位信息）和 `analysis`（分析依据）
- 不做空泛评价，不写"值得点赞"等 AI 味评语
- 权重可在 YAML 配置中调整，支持按 profile 切换
- **原文证据（excerpts）在 Map 阶段产出**（Map 看得到原文，能逐字摘录），Reduce 阶段只引用指针 `{chapterId, excerptIndex}`，不再产逐字文本（v2.2 修订，见下方「原文证据机制」）
- `emotionalCurve` 只保留 `tension`（valence 删掉，Map 不产出）
- Chapter `id` 改为 string 类型（支持 "vol1-ch003" 等），增加 `kind` 字段

### 原文证据机制（v2.2 修订：excerpts 从 R2 下放到 Map）

> **原设计缺陷（spike 发现）：** v2.1 要求 R2（五维评分）逐字摘录原文作为 quotes。但 R2 的输入只有 Map 的摘要，**看不到章节原文**，无法逐字摘录——spike 实测 10 条 quote 全是基于摘要的改写概述，回链命中率 0%。这不是 prompt 问题，是架构问题：基于摘要的全局评判与"逐字引用原文"在信息层面矛盾。

**修订（方案 A）：职责彻底切分——Map 产证据，Reduce 引指针。**

**数据流：**

```
Map（每章，看得到原文）
  产出: excerpts[] = [
    { text: "逐字摘录的原文片段", dimension: "characterization",
      reason: "为什么这条能代表该章在此维度的表现" },
    ...每章 3–5 条，覆盖至少 2 个不同维度
  ]
  → 后端回链: 在本章正文里 indexOf(text) → 补 offset
    （原文在手，精确/模糊命中率高）

Reduce R2（基于全部摘要做全局评判，看不到原文）
  产出: dimensions.*.analysis 里引用证据时
       只在文本中嵌入指针标记 [chapterId#excerptIndex]
       （从 Map 的 excerpts 池里指一条）
  → 前端展示时解析指针 → 去 Map 结果取 excerpt（已带 offset）
```

**为什么这样可行：**
1. Map 阶段天然能逐字摘录（输入就是章节原文），摘录质量有保障。
2. excerpts 标注 `dimension`（代表该章在哪个维度的表现），R2 评判某维度时能从池里筛相关证据。
3. 回链只在 Map 阶段做一次（每章正文在手），命中率远高于让 R2 凭摘要猜原文。
4. R2 的 analysis 写起来更自然——说"人物塑造见 ch001 母亲克制反应、ch005 小宇童真发问"即可，用指针引用，不必硬背原文措辞。

**回链实现**（`src/core/quote-linker.ts`，纯逻辑、可单测，spike 已验证）：
1. 在 excerpt 所属章节正文里**精确字符串匹配** `indexOf(text)` → `offset = 命中位置`。
2. 精确失败 → **归一化模糊匹配**（去空白、全角标点转半角后比对）→ 近似 offset。
3. 仍失败 → `offset = null`，前端降级为滚动到该章 + 高亮 `reason` 文本。
- `offset` 定义：该章正文中的字符偏移（UTF-16 code unit）。`Excerpt.offset: number | null`。

**Map excerpts 数量约束**：每章 3–5 条，覆盖至少 2 个不同维度。过少则 R2 证据不足；过多则 Map token 膨胀（spike 实测每条 excerpt 约 +60 token，5 条约 +300 token/章，可接受）。

**R2 analysis 的指针格式**：analysis 是自然语言文本，在其中嵌入 `[ch001#2]`、`[ch003#0]` 这类指针标记（chapterId#excerptIndex）。前端解析标记 → 取对应 excerpt → 渲染为可点击的原文引用块。后端校验指针有效性（chapterId 存在、excerptIndex 在范围内），无效指针降级为纯文本。

### 小说特有检查项（在 suggestions 中体现，不单独加维度）

以下检查项融入 Reduce 阶段的 suggestions 输出：

- **世界观/设定一致性**：跨章节检测设定矛盾（如角色能力变化、地理描述不一致）
- **视角/叙事一致性**：检测视角跳跃或叙述者身份混乱
- **信息密度/注水检测**：识别低信息密度段落（大段环境描写无情节推进、重复表达）
- **章末钩子/追读力**：评估每章结尾是否设置了悬念或钩子，追读力强弱

## 二·补、评估 Prompt 设计（结构骨架 + 评分标尺）

> 本节给出 prompt 的**结构骨架与评分标尺（rubric）**，不写 prompt 全文。prompt 全文留待 spike 阶段用真实小说 3–5 章调优后落库（`config/prompts/*.md`）。Prompt 的所有改动都要跑 golden case 回归（见「测试策略」）。

### 设计原则

1. **职责分离**：Map 做"读懂单章 + 摘录证据"，Reduce 做"评价全书 + 引用证据"。Reduce 看不到原文，所以**逐字摘录的职责归 Map**（v2.2 修订）。
2. **强制结构化输出**：所有 prompt 都带 JSON Schema 契约，输出经 `json-validator.ts` 校验。**注意（spike 发现）：GLM 兼容端点的 `output_schema` 字段被接受但不强制约束**，必须靠 `parseJSONRobust` 容错解析 + schema 校验 + 重试三重保险（见「JSON 稳定性设计」）。
3. **稳定前缀可 cache**：system + 评估规则 + few-shot 放请求前缀（命中 prompt cache）；逐章正文/摘要放末尾（每章不同，不进 cache）。详见「Prompt Caching 设计」。
4. **证据必逐字**：Map 产出的 excerpts 必须**从本章原文逐字复制**，由后端回链定位 offset（见「原文证据机制」）。Reduce 引用证据时只输出指针 `[chapterId#excerptIndex]`。

### Map Prompt 结构

```
[System]  （稳定前缀，进 prompt cache）
  - 角色：资深小说编辑，负责单章细读
  - 输出契约：严格按 JSON Schema 输出，禁止解释性前缀/后缀
  - Few-shot：1 个高质量章节摘要 + excerpts 示例
[User]
  - 评估规则片段（摘要/张力/事件/人物/excerpts 的口径定义）
  - {单章正文}                       ← 末尾，每章不同
  - "请输出 JSON：{summary, emotionalTension, keyEvents, characters, excerpts}"
```

**Map 输出契约**（v2.2：5 项）：
- `summary`(本章梗概, 80–200 字) / `emotionalTension`(0–100) / `keyEvents`(2–5 条) / `characters`(本章出场角色名数组)
- **`excerpts`**（3–5 条，逐字摘录的原文证据）：每条 `{ text, dimension, reason }`
  - `text`：**必须从本章正文逐字复制**，10–80 字，不改写不增删
  - `dimension`：该片段代表的维度（storyStructure/characterization/writingQuality/emotionalResonance/marketPotential 之一）
  - `reason`：一句话说明为什么这条能代表该章在此维度的表现（供 R2 筛选证据 + 前端兜底展示）

### Reduce Prompt 结构（按 R1–R4 子调用分别构造，见「阶段三：Reduce」）

每个子调用独立 prompt、独立 system、独立输出 schema：

```
[R1 人物归一化]
  System: 角色=人物谱系编辑；输出=归一化后的 characters[]
  User:   全部章节的 characters[] 出场记录
  出:     合并同名/昵称/代称后的统一角色列表（含 role/firstAppearance）

[R2 五维评分]   ← 价值核心
  System: 角色=总编，基于全书做宏观评判；附五维评分标尺（见下）
  User:   章节摘要序列 + 情绪张力序列 + R1 的人物列表
          + 各章 excerpts 清单（含 chapterId/index/dimension/reason）
          + 评估规则 + profile 权重
  出:     dimensions（五维，各含 score/subscores/analysis）
          analysis 中用 [chapterId#excerptIndex] 指针引用 Map 的 excerpts 作证据

[R3 情绪曲线]
  System: 角色=叙事节奏分析师
  User:   章节情绪张力序列（来自 Map）
  出:     emotionalCurve（可标注波峰/波谷/拖沓段）

[R4 改进建议]
  System: 角色=改稿指导，输出"手术刀式"可执行建议
  User:   R2 维度评分 + 章节摘要 + R1 人物 + excerpts 清单 + 规则中的小说特有检查项
  出:     suggestions[]（每条含 dimension/type/content/relatedChapters/excerptRef）
          excerptRef = {chapterId, excerptIndex} 指针，指向 Map 的某条 excerpt
```

### 五维评分标尺（Rubric，防分数漂移）

> 评分必须落到锚点描述的区间内。LLM 在给出 score 时需内化下表；每维度附 1 个 few-shot 高质量引用（放 system，命中 cache）。下表为骨架，spike 阶段细化。

| 维度 | 90+ (S) | 60–75 (B) | <50 (D) |
|------|---------|-----------|---------|
| **故事架构** | 结构完整、起承转合清晰、节奏精准、悬念层层递进 | 结构基本成立、有起伏但局部拖沓或断裂 | 主线混乱、重大逻辑漏洞、节奏失控 |
| **人物塑造** | 角色立体、弧光完整、动机自洽、关系网络有张力 | 角色可辨识、有基本弧光但动机偶有牵强 | 脸谱化、动机缺失、弧光断裂 |
| **文笔质量** | 语言精炼有个人风格、对话鲜活、修辞得当 | 基本通顺但平淡、对话偏书面/生硬 | 语病频出、风格混乱、表达吃力 |
| **情感共鸣** | 代入感强、情绪曲线有层次、共鸣深 | 有情感起伏但代入一般、情绪推进线性 | 情感扁平、难以共情 |
| **市场潜力** | 类型定位清晰、受众明确、有差异化卖点 | 定位模糊但可归类、差异化不足 | 类型混乱、受众不明、无差异化 |

> 评分禁止：空泛评价（"值得点赞"/"很有潜力"）、无引用的论断、AI 味套话。每条 analysis 必须引用原文证据（quotes）。

### Prompt 版本管理

- prompt 落库 `config/prompts/{map,reduce-r1,reduce-r2,reduce-r3,reduce-r4}.md`，带 frontmatter `version`。
- 评估结果 JSON 的 `task.configSnapshot` 记录 prompt 版本哈希，保证可复现/可回归。

## 三、评估流程（Map-Reduce 架构）

### 阶段一：解析 + 分章

**策略：结构解析优先，LLM fallback。**

```
文件上传 → 文档解析(统一接口) → 提取纯文本 → 结构化分章
  输出：[{ id: "ch001", title, content, startIndex, endIndex, kind }]
```

分章按优先级逐级尝试，上一级成功即跳过后续：

| 优先级 | 策略 | 适用格式 | 实现方式 |
|--------|------|---------|---------|
| 1 | 内置章节结构 API | EPUB/DOCX | 直接读取文档章节元数据，零成本 |
| 2 | 正则匹配 | TXT/网文 | 匹配"第X章/回/节" + 空行分隔模式 |
| 3 | 启发式分页 | PDF | 按页/段落启发式切分 |
| 4 | LLM 辅助 | 所有格式（fallback） | 仅对以上方法失败的区间调用 LLM，不是全书 |

> MVP 阶段仅实现正则分章（TXT），LLM 分章放 v2。

### 阶段二：Map — 逐章评估（独立，无前章依赖）

```
章节列表 → 并发调用 GLM API（默认 5 并发）
  每章输入：章节正文 + 评估规则 YAML
  每章输出：{
    summary: 章节摘要,
    emotionalTension: 本章情绪张力值(0-100),
    keyEvents: 关键事件列表,
    characters: 本章出场角色,
    excerpts: [{ text, dimension, reason }]  ← v2.2 新增：逐字摘录的原文证据
  }
  每章完成后：后端对 excerpts 回链定位 offset（见「原文证据机制」）
```

**关键设计决策：**
- **Map 产出 5 项**（v2.2：从 4 项增至 5 项）：摘要、情绪张力、关键事件、出场角色、**原文证据 excerpts**。五维评分仍在 Reduce 阶段计算。
- **excerpts 为何放 Map**（spike 发现）：Reduce 看不到原文，无法逐字摘录（spike 实测命中率 0%）。Map 输入就是原文，天然能逐字摘录，回链命中率高。
- **并发 Map 取消前章摘要依赖**：每章独立评估，跨章衔接全部交给 Reduce。并发时不传递前章摘要（因为并行时前章可能还没完成）。
- 并发控制：默认 5 并发，可配置。带任务级和单章级预算控制。

**Checkpoint 机制：**
Map 阶段每完成一章即将结果（含已回链的 excerpts）写入 partial JSON。支持中途查看已完成部分，也支持中断恢复。

```typescript
interface EvaluationCheckpoint {
  phase: 'split' | 'map' | 'reduce'
  completedChapterIds: string[]
  reduceSubstep?: 'r1' | 'r2' | 'r3' | 'r4' | 'done'  // v2.1：Reduce 子步骤进度
  partialResultPath: string
  sessionId?: string
}
```

### 阶段三：Reduce — 全局分析（修订：Reduce Pipeline 多子调用）

**默认方式：Reduce Pipeline（R1→R2→R3→R4 四个子调用串联），全部走 GLM API 直调。**

> **为什么从"单次调用"改为多子调用：** 原设计论证了"输入 ~100K token < 200K 上下文"，但**只防了输入超限，没防输出超限**。Reduce 一次要产出五维详细分析 + 人物归一化 + 情绪曲线 + 全部 suggestions，输出可达数十 K token，易撞输出上限中途截断。拆成多个小调用，每个调用的输出都控制在限额内，且可独立重试、独立计费、独立 checkpoint 恢复。
>
> **spike 验证（v2.2）**：R2 单独跑实测输出 1986 token，远未逼近上限。对 ≤200 章常规小说 R2 单次调用足够；树形 Reduce（>200章）仍保留作极端兜底。

```
Map 全部章节结果（含已回链的 excerpts）
   │
   ├─ R1 人物归一化  → characters[]（合并同名/昵称/代称）
   │     输入: 全部章节 characters[] 出场记录
   │     输出: 统一角色列表（role / firstAppearance / keyChapters / relationships）
   │
   ├─ R2 五维评分    → dimensions（价值核心）
   │     输入: 章节摘要序列 + 情绪张力序列 + R1 人物
   │           + 各章 excerpts 清单（chapterId/index/dimension/reason）  ← v2.2：作证据池
   │           + 评估规则 + profile 权重
   │     输出: 五维 score/subscores/analysis
   │           analysis 用 [chapterId#excerptIndex] 指针引用 excerpts（v2.2：不再产逐字 quotes）
   │
   ├─ R3 情绪曲线    → emotionalCurve
   │     输入: 章节情绪张力序列（来自 Map）
   │     输出: 整体情绪节奏曲线（可标注波峰/波谷/拖沓段）
   │
   └─ R4 改进建议    → suggestions[]
         输入: R2 维度评分 + 章节摘要 + R1 人物 + excerpts 清单 + 规则中的小说特有检查项
         输出: 跨章改进建议（含 dimension/content/relatedChapters/excerptRef）
               excerptRef = {chapterId, excerptIndex} 指针（v2.2：替代原 quote 字段）
```

**子调用依赖与降级：**
- R1 无依赖，最先跑；R2/R4 依赖 R1 的人物列表；R3 无依赖，可与 R1 并行。
- **独立重试**：每个子调用失败 → 带 schema 错误信息重试（最多 2 次）→ 仍失败按严重度降级：
  - R2 失败 = **致命**（无五维评分则报告无意义）→ 整任务 `failed`。
  - R1/R3/R4 失败 = **非致命** → 报告照常生成，对应区块标注"本项分析失败/缺失"。
- **独立计费**：每个子调用 cost 单独累计进 `task.cost`，超 `taskMaxBudgetRmb` 则整任务暂停（v2.2：预算改人民币计价）。

**人物归一化（R1）：**
用 LLM 合并同名/昵称/代称（如 "张三"/"张 三"/"三哥"），输出统一角色列表，供 R2/R4 引用一致的角色标识。

> Claude Code 的 `mcp serve` 是暴露给外部 MCP Client 用的，不做会话编排。Reduce 各子调用保留 `stream-json` stdin/stdout 作为 Claude Code fallback 的会话方式，但默认走 Anthropic API 直调。

**树形 Reduce（章节数 > 200 时）：**
仅作用于 R2/R4 的**输入侧**——先每 ~20 章 Local Reduce 生成弧摘要压缩输入量，再 Global Reduce 汇总。R1/R3 不受影响（人物出场和情绪张力序列本身规模可控）。

```
[ch1-ch20]   → Local Reduce → 弧摘要1
[ch21-ch40]  → Local Reduce → 弧摘要2
...
[ch181-ch200]→ Local Reduce → 弧摘要10
    ↓
[弧摘要1..10] → R2/R4 Global Reduce → 最终全局分析与建议
```

### 评分聚合公式（修订：消除章级加权矛盾）

> **关键决策：Map → 证据，Reduce → 维度分，代码 → 总分。**
> 五维每一维的维度分（0–100）由 Reduce 基于全书摘要**直接给出一个综合分**，**不做章级机械加权**。理由：
> 1. Map 不产出五维评分（市场潜力无法按章评估，其余维度按章评也不准确）。
> 2. LLM 的全局判断优于机械加权——能处理"前 50 章优秀、后 30 章崩盘"这类非线性情况，加权会平均化掩盖问题。
> 3. Map 产出的摘要/情绪张力/关键事件/人物作为 Reduce 的**输入证据**，由 LLM 在判分时综合权衡，不参与数值聚合。
>
> 因此**删除原"章级 subscore → 维度分"加权公式**，避免与"五维全部在 Reduce 产出"自相矛盾。

代码只做两步机械聚合（确定性、可单测）：

```typescript
// 1. 维度分（Reduce 产出）→ 总分（代码聚合）
overall.totalScore = round(Σ(dimensionScore × profileWeight))

// 2. 总分 → 等级（代码查表）
grade = lookupGrade(overall.totalScore, yamlThresholds)
```

- `dimensionScore`：Reduce R2 子调用产出，每维一个 0–100 整数（详见「阶段三：Reduce」）。
- `profileWeight`：取自当前 profile（default/revision/submission），合计为 1.0。
- `round`：总分四舍五入到整数；查表用 `>=` 阈值（S≥90, A≥80, B≥70, C≥60, D<60）。

```yaml
# config/default.yml
gradeThresholds:
  S: 90
  A: 80
  B: 70
  C: 60
  D: 0
```

### 错误处理与重试

- **单章评估失败** → 重试 1 次（JSON Schema 校验失败 → 自动重试，最多 2 次）→ 仍失败则标记跳过，最终报告标注缺失章节
- **Reduce 阶段中断** → 可基于已有 checkpoint 恢复，不需要重新 Map
- **LLM 输出校验**：强制 JSON Schema 校验，失败自动重试（最多 2 次），超过则报错

### 成本与超时控制

```typescript
interface CostConfig {
  // 任务级预算（v2.2：人民币计价，因引擎改 GLM）
  taskMaxBudgetRmb: number;       // 默认 10（GLM 成本极低，实测 20 万字约 1.3 元）
  // 单章预算
  perChapterMaxBudgetRmb: number; // 默认 0.1
  // 动态超时：baseMs + wordCount * factor
  timeout: {
    baseMs: number;               // 默认 30000
    perThousandWordsMs: number;  // 默认 200
  };
  // watchdog：流空闲超时
  streamIdleTimeoutMs: number;   // 默认 90000 (90s)
}
```

- watchdog：流空闲 90s 无新事件 → kill 当前调用
- 单次调用超时：`30000ms + wordCount * 0.2ms/千字`
- 任务级总预算硬限制：¥10（可配置，spike 实测 20 万字仅 ¥1.3，留充足余量）

## 四、统一抽象层

### 核心接口

```typescript
interface AIAgentEvent {
  type: 'assistant' | 'tool_use' | 'result' | 'error';
  content: string;
  data?: any;
  usage?: { inputTokens: number; outputTokens: number; costRmb: number };  // v2.2：人民币计价
}

interface AIAgentAdapter {
  readonly name: string;

  // 单次调用
  run(prompt: string, options: RunOptions): AsyncIterable<AIAgentEvent>;

  // 会话模式（Claude Code fallback via stream-json stdin/stdout）
  createSession(options?: SessionOptions): AIAgentSession;

  // 查询可用性
  isAvailable(): Promise<boolean>;
}

interface AIAgentSession {
  send(prompt: string): AsyncIterable<AIAgentEvent>;
  close(): void;
}

interface RunOptions {
  cwd?: string;
  systemPrompt?: string;
  allowedTools?: string[];
  disallowedTools?: string[];
  maxBudgetRmb?: number;   // v2.2：人民币计价
  model?: string;
  timeoutMs?: number;
  outputSchema?: object;   // ⚠️ spike 发现：GLM 兼容端点接受此字段但不强制约束，须配合 parseJSONRobust + 校验
}

interface SessionOptions {
  cwd?: string;
  systemPrompt?: string;
  model?: string;
}
```

### Adapter 实现（v2.2：GLM 默认）

| Adapter | `run()` 底层 | `createSession()` 底层 | 优先级 |
|---------|-------------|----------------------|--------|
| **`BigModelAdapter`** | 智谱 GLM，经 Anthropic 兼容端点 `/v1/messages`（裸 fetch，无 SDK） | — | **默认** |
| `AnthropicAPIAdapter` | Anthropic SDK（`messages.create`） | — | 备选（有原生 Anthropic key 时） |
| `ClaudeCodeAdapter` | `claude -p --output-format stream-json` | stream-json stdin/stdout | fallback |

> **BigModelAdapter 为默认实现**（v2.2 修订，源自 spike）。理由：
> 1. 实测成本极低——20 万字小说约 **¥1.3/本**，比 Anthropic Sonnet 低 30-50 倍。
> 2. 智谱提供 Anthropic 兼容端点 `https://open.bigmodel.cn/api/anthropic`，调用方式与 Anthropic 几乎一致，`BigModelAdapter` 复用 `/v1/messages` 协议，仅改 base_url + 认证头 + 模型名。
> 3. glm-5.2 支持 1M 上下文，长篇评估无压力。
> 4. spike 已验证：Map 5 章稳定产出、R2 五维评分质量高且无截断。
>
> **认证**：`Authorization: Bearer <key>` + `x-api-key`（兼容端点任一可用）。环境变量 `ANTHROPIC_AUTH_TOKEN` 或 `ZHIPUAI_API_KEY`。

### 引擎配置（engines.yml）

```yaml
# v2.2：GLM 为默认引擎
default: bigmodel
fallback: claude-code

bigmodel:
  baseUrl: https://open.bigmodel.cn/api/anthropic
  model: glm-5.2            # 旗舰；开发调试可用 glm-4.7-flash（免费）
  maxBudgetRmb: 10
  perChapterMaxBudgetRmb: 0.1
  # prompt caching 由兼容层自动处理（隐式前缀匹配），无需 cache_control
  # ⚠️ output_schema 字段被接受但不强制，须配合 parseJSONRobust + 校验（见 10.6）

anthropic-api:              # 备选：有原生 Anthropic key 时
  model: claude-sonnet-4-20250514
  maxBudgetRmb: 70          # 按 $10 汇率换算
  perChapterMaxBudgetRmb: 1.5
  promptCaching: true       # Anthropic 原生支持显式 cache_control

claude-code:                # fallback：本地无任何 API key 但装了 Claude Code
  model: claude-sonnet-4-20250514
  maxBudgetRmb: 5
  allowedTools: [Read, Glob, Grep]
```

### 关键设计决策

- 所有 adapter 实现同一接口，上层代码只依赖 `AIAgentAdapter`
- `run()` 返回 `AsyncIterable`，支持流式处理（进度条、实时更新）
- 评估场景只开放只读工具（Read、Glob、Grep），禁止文件写入
- 超时和预算双重兜底，防止失控
- **LLM 输出三重保险**（v2.2，源自 spike）：`parseJSONRobust` 容错解析 → schema 校验 → 重试（最多 2 次）。详见 10.6/10.7。
- **prompt caching 差异**：Anthropic 需显式 `cache_control`；GLM 兼容端点自动前缀匹配，无需任何 cache 代码（保持 system 前缀稳定即可）。

### 各格式实现

## 九、关键类型

```typescript
type TaskStatus =
  | 'pending' | 'parsing' | 'splitting'
  | 'mapping' | 'reducing' | 'generating'
  | 'completed' | 'failed'
// 注：'reducing' 覆盖 Reduce Pipeline 的 R1→R2→R3→R4 四个子步骤，
//     细粒度进度见 EvaluationCheckpoint.reduceSubstep（不改本枚举，避免状态爆炸）

interface EvaluationTask {
  id: string
  filePath: string
  fileName: string
  format: 'txt' | 'epub' | 'pdf' | 'docx'
  status: TaskStatus
  progress: { current: number; total: number; message: string }
  error: Error | null
  engine: string
  configSnapshot: object          // 含 prompt 版本哈希、profile、权重，保证可复现
  cost: { inputTokens: number; outputTokens: number; totalRmb: number }  // v2.2：人民币计价
  checkpoint: EvaluationCheckpoint | null
  sourceWordCount: number
  chapterCount: number
  createdAt: Date
  completedAt?: Date
  resultFile?: string
  reportFile?: string
}

interface Chapter {
  id: string                  // string 类型，支持 "vol1-ch003" 等
  title: string
  kind: 'main' | 'extra' | 'prologue' | 'epilogue'
  wordCount: number
  summary: string
  keyEvents: string[]
  characters: string[]
  emotionalTension: number   // 0-100
  excerpts: Excerpt[]        // v2.2 新增：Map 产出的原文证据（已回链 offset）
}

interface EvaluationCheckpoint {
  phase: 'split' | 'map' | 'reduce'
  completedChapterIds: string[]          // Map 阶段已完成章节
  reduceSubstep?: 'r1' | 'r2' | 'r3' | 'r4' | 'done'  // Reduce 阶段子步骤进度，支持 R 子步骤级恢复
  partialResultPath: string
  sessionId?: string
}

/** v2.2：原文证据（Map 产出，后端回链 offset）。替代 v2.1 的 Quote。 */
interface Excerpt {
  text: string               // 逐字摘录的原文片段（Map 阶段从本章正文复制）
  dimension: DimensionKey    // 该片段代表的维度
  reason: string             // 为什么这条代表该维度（供 R2 筛选 + 前端兜底）
  chapterId: string
  offset: number | null      // 后端回链补全；精确/模糊匹配均失败时为 null（见原文证据机制）
}

/** v2.2：Reduce 阶段引用证据的指针（R2 analysis 内嵌 / R4 excerptRef）。指向某章的某条 excerpt。 */
interface ExcerptRef {
  chapterId: string
  excerptIndex: number       // 该章 excerpts 数组的下标
}

type DimensionKey =
  | 'storyStructure' | 'characterization' | 'writingQuality'
  | 'emotionalResonance' | 'marketPotential'

interface CharacterRelationship {
  target: string
  type: string
  strength: number
}
```

## 十、输出 Schema 契约（用于 LLM `outputSchema` 校验）

> 以下 JSON Schema 直接用于 `RunOptions.outputSchema` 与 `json-validator.ts`。LLM 输出必须匹配；校验失败自动重试（最多 2 次）。字段约束尽可能严格（范围、必填、数组长度上限），降低 LLM 漂移空间。

### 10.1 Map 输出 Schema（单章）

```jsonc

  "type": "object",

  "additionalProperties": false,
  "properties": {
    "summary":        { "type": "string", "minLength": 40, "maxLength": 400 },
    "emotionalTension": { "type": "integer", "minimum": 0, "maximum": 100 },
    "keyEvents":      { "type": "array", "items": { "type": "string" }, "minItems": 1, "maxItems": 8 },

    "excerpts": {                          // v2.2 新增：逐字摘录的原文证据
      "type": "array",
      "minItems": 3,
      "maxItems": 5,
```

### 10.6 JSON 稳定性设计（v2.2 新增，源自 spike 发现）

> **spike 发现：** GLM 兼容端点的 `output_schema` 字段被接受但**不强制约束输出**。glm-5.2 在生成中文内容时，常在 JSON 字符串值内用**未转义的 ASCII 双引号**包裹对话（如 `summary: "...只说"再见"..."`），破坏 JSON 结构。spike 初次跑 5 章有 2 章因此解析失败。

**三重保险（按执行顺序）：**

1. **容错解析**（`src/json-util.ts` 的 `parseJSONRobust`，spike 已实现验证）：
   - 直接 `JSON.parse` → 去markdown包裹 → 提取 `{...}` → **修复字符串值内未转义引号**（逐字符扫描，判断引号是结构符还是内容符）
   - spike 实测：该解析器一次性修复了所有失败用例，重试机制未被触发
2. **schema 校验**：`json-validator.ts` 按字段约束（类型/范围/必填/数组上限）校验
3. **重试**：校验失败 → 把错误明细拼回 prompt 重试（最多 2 次）

> 设计文档原 v2.1 修订12 只提"JSON Schema 校验+重试"，**漏了最关键的容错解析层**。容错解析是第一道防线，能就地修复大多数格式问题，比重试省 token、省时间。

### 10.7 校验 + 重试 + 降级流程

```
LLM 输出
  → parseJSONRobust(text)               ← 第一道：容错解析（含未转义引号修复）
     ├─ 解析失败 → 进入重试
     └─ 解析成功 → json-validator.validate(output, schema)  ← 第二道：schema 校验
                   ├─ 通过 → Map excerpts 回链补 offset / R2 指针校验 / 聚合
                   └─ 失败 → 进入重试
  → 重试（最多 2 次，把错误明细拼回 prompt，降温）
                 ├─ 仍失败 & 是 R2（致命）→ 整任务 failed，写错误状态
                 ├─ 仍失败 & 是 Map 单章 → 标记该章 skipped，报告标注缺失章节
                 └─ 仍失败 & 是 R1/R3/R4（非致命）→ 该区块标注"分析缺失"，继续生成报告
```

### 实测数据（spike，glm-5.2，5 章样本）

| 阶段 | input token | output token | 费用 |
|------|------------|-------------|------|
| Map（5 章并发） | 5270 | 1143 | ¥0.064 |
| R2 五维评分（单次） | 2208 | 1986 | ¥0.042 |
| **合计（5 章）** | 7478 | 3129 | **¥0.106** |

### 外推到 20 万字小说（约 80 章，按 2500 字/章）

| 阶段 | 估算 token | 估算费用 |
|------|-----------|---------|
| Map（80 章，含 excerpts） | in 134K + out 18K | ¥1.52 |
| R1 人物归一化 | in 30K + out 3K | ¥0.33 |
| R2 五维评分 | in 23K + out 2K | ¥0.25 |
| R3 情绪曲线 | in 10K + out 2K | ¥0.12 |
| R4 改进建议 | in 35K + out 5K | ¥0.40 |
| **合计** | — | **约 ¥2.6/本** |

> 注：spike 样本每章 570 字偏短，真实网文每章 2500 字使 Map input 增大约 4 倍；含 excerpts 后每章 output 增约 300 token。综合修正后 20 万字约 **¥2.6/本**，仍极低。

### 成本优化策略

1. **Prompt Caching（GLM 自动）**：GLM 兼容端点隐式前缀匹配，无需 `cache_control` 代码。保持 system + 评估规则 + few-shot 前缀稳定即可自动命中，进一步降低 Map 输入成本。
2. **Flash 模型开发期**：开发调试用 `glm-4.7-flash`（免费，200K 上下文），上线切 `glm-5.2` 保证质量。
3. **并发效率**：5 并发 Map 可缩短总时间 ~60%，减少等待成本。

### Prompt Caching 设计（v2.2：GLM 隐式缓存）

> **v2.1 原设计基于 Anthropic 显式 `cache_control`，v2.2 改 GLM 后简化为隐式缓存。**

**cache 布局（Map 每章请求）：**
```
[System block]                                ✅ GLM 自动缓存（每章前缀相同）
  ├ 角色定位
  ├ JSON Schema 契约
  ├ 评估规则片段
  └ Few-shot 示例
[User block]                                  ❌ 不进 cache（每章不同）
  └ {本章正文}
```
- **GLM 兼容端点无需 `cache_control` 字段**——只要 system 前缀稳定，自动前缀匹配命中缓存。
- spike ping 测试已确认 usage 里有 `cache_read_input_tokens` 字段（计量存在）；正式开发时做 A/B 对比量化命中率。
- Reduce 各子调用（R1–R4）的 system + 规则部分同样稳定，长任务内可命中。

> **与 Anthropic 的差异**：Anthropic 需显式打 `cache_control: {type: "ephemeral"}`（最多 4 断点）；GLM 零代码自动缓存。`BigModelAdapter` 比 `AnthropicAPIAdapter` 实现更简单。

### 成本模型（简化）

```
Map 成本 ≈ 章节数 × 平均每章 token × 单价（glm-5.2 ≈ ¥10/百万 token）
  - 20万字 ≈ 80章 × ~1700 input + ~480 output token/章
Reduce 成本 ≈ R1+R2+R3+R4 各子调用之和
总计 ≈ ¥2-3/本（glm-5.2）；开发期用 glm-4.7-flash 可降至接近 ¥0
```

### 任务预算控制

```yaml
# engines.yml（v2.2：人民币计价）
taskMaxBudgetRmb: 10          # 整次评估任务上限（实测 ¥2.6，留 4 倍余量）
perChapterMaxBudgetRmb: 0.1  # 单章评估上限
```

## 十二、安全设计

| 风险 | 防护措施 |
|------|---------|
| 文件上传路径穿越 | 文件存 `data/novels/{uuid}.{ext}`，不使用原始文件名 |
| 恶意文件（zip bomb 等） | 扩展名白名单校验 + 文件大小上限 |
| HTML 报告 XSS | 所有动态内容 HTML escape；JSON 用 `<script type="application/json">` 内嵌 |
| 本地 Web 服务暴露 | 绑定 `127.0.0.1`；若需局域网访问，加 token / basic auth |
| 数据泄露 | 数据保留策略：默认 30 天自动清理；README 提醒用户 |
| API 费用失控 | 任务级 + 单章级双重预算上限，超额暂停并写入任务状态 |

## 十三、测试策略（修订：补全原缺失的测试计划）

> 原设计有 `tests/` 目录但零字提及测试。13 天交付不带测试计划风险大。以下为分层测试策略。

### 13.1 纯逻辑单元测试（无 LLM，必须 100% 覆盖核心算法）
- **分章正则**：各种章节标题模式（"第X章/回/节"、"Chapter X"、"卷X"）、边界（无章节标志的全文、章末粘连）、编码混用。
- **编码检测**：GBK/GB18030/UTF-8/UTF-16/BOM 检测准确性（chardet 包装层）。
- **quotes 回链匹配**：精确命中、空白/标点差异的模糊命中、完全不匹配返回 null（这是本次新增的核心逻辑，重点测）。
- **成本/预算计算**：token→USD 换算、单章/任务级预算超额判定与暂停。
- **overall 聚合**：维度分 × 权重 → 总分 → 等级查表（含权重边界、并列等级阈值）。

### 13.2 集成测试（mock LLM adapter）
- **Checkpoint 恢复**：Map 跑到中途中断 → 重启从 `completedChapterIds` 恢复，不重跑已完成章。
- **Reduce Pipeline 串联**：R1→R2→R3→R4 依次执行、子步骤状态推进、`reduceSubstep` 正确更新。
- **降级路径**：R2 失败→任务 failed；R1/R3/R4 失败→报告标注缺失但生成；Map 单章失败→跳过并标注。
- **JSON 校验重试**：故意返回坏 JSON，验证重试 2 次后正确降级。

### 13.3 LLM 评估质量测试（golden case 回归）
- 准备 **3–5 本已知质量的小说**（覆盖不同类型：玄幻/言情/悬疑/纯文学），人工标注每维度期望评分区间（如 ±10 分容差）。
- CI 跑分需落在区间内；prompt 改动后若评分漂移出区间，CI 失败，强制人工 review。
- golden case 数据与期望值入仓 `tests/golden/`，标注版本。

### 13.4 LLM 录制回放（防 prompt 调整破坏 schema）
- 用 VCR/fixture 录制真实 LLM 响应（按 prompt 版本哈希索引）。
- prompt 文案微调时，用录制响应做回归比对——验证 schema 仍能解析、关键字段不丢。
- 真实 API 调用仅在 golden case 回归或 spike 时触发，日常 CI 走录制回放（省 token、稳定）。

### v2.2 spike 验证后修订（本次）

> 基于 prompt spike 实测结果（详见 [docs/spike/2026-07-08-spike-report.md](../spike/2026-07-08-spike-report.md)）的设计修订。spike 四个验证点：Map 稳定✓、R2 不截断✓、quotes 回链 0%✗、成本 ¥1.3/本✓。来源标注：spike 发现(S)。

#### P0 架构修正（spike 暴露的缺陷）

| # | 修正内容 | 来源 |
|---|---------|------|
| 35 | **quotes 机制重构（方案A）**：原 R2 逐字摘录 → 改为 Map 产 excerpts（逐字摘录+维度标注+回链 offset）、R2 用 `[chapterId#excerptIndex]` 指针引用。根因：R2 看不到原文，spike 实测命中率 0% | S |
| 36 | **引擎层 GLM 化**：默认 `AnthropicAPIAdapter` → `BigModelAdapter`（智谱 GLM 经 Anthropic 兼容端点）。成本降 30-50 倍，调用方式几乎不变 | S |
| 37 | **JSON 稳定性三重保险**：新增 `parseJSONRobust` 容错解析（修复中文未转义引号）作为第一道防线，原 v2.1 只提"schema 校验+重试"不够 | S |

#### P1 规格修正

| # | 修正内容 | 来源 |
|---|---------|------|
| 38 | **Map 产出从 4 项增至 5 项**：+excerpts（3-5 条/章，逐字摘录+dimension+reason）。Map schema 10.1 更新 | S |
| 39 | **R2 schema 去 quotes**：dimension 只保留 score/subscores/analysis，analysis 内嵌指针引用 excerpts（10.3 更新） | S |
| 40 | **R4 quote 字段改 excerptRef 指针**：`{chapterId, excerptIndex}` 替代逐字文本（10.5 更新） | S |
| 41 | **预算改人民币计价**：`maxBudgetUsd`→`maxBudgetRmb`、`totalUsd`→`totalRmb`、`costUsd`→`costRmb`。默认 taskMaxBudgetRmb=10 | S |
| 42 | **成本估算换实测数据**：20 万字 ¥2.6/本（原 Anthropic 预估 35-70 元）。Prompt caching 改 GLM 隐式缓存（删 cache_control 代码） | S |
| 43 | **output_schema 不可依赖**：标注 GLM 兼容端点接受该字段但不强制约束，须配合容错解析+校验 | S |
