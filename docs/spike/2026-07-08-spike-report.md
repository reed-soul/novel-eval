## 二、验证点①：Map 阶段输出稳定性 ✅

**设置**：5 章并发，每章一次调用，`output_schema` 强约束 + 容错解析器 + 重试（最多3次）。

**结果**：5/5 章全部通过 schema 校验，重试 0 次。

| 章节 | 标题 | 情绪张力 | 出场角色 | input tok | output tok |
|------|------|---------|---------|-----------|------------|
| ch001 | 归乡 | 28 | 陈默、母亲 | 1007 | 215 |
| ch002 | 旧人 | 55 | 陈默、林晚、母亲 | 1056 | 205 |
| ch003 | 决定 | 38 | 陈默、母亲 | 1048 | 236 |
| ch004 | 重逢 | 48 | 陈默、林晚、小宇 | 1088 | 228 |
| ch005 | 春天 | 38 | 陈默、林晚、小宇 | 1071 | 259 |

**质量评估**：
- 情绪张力曲线合理（重逢旧人 ch002 达到峰值 55，符合情节）
- 人物识别准确（过场角色"出租车司机"因无名字未被列入，符合 prompt 设计）
- 摘要客观陈述，无评价词

**附：JSON 稳定性问题（重要发现）**

初次运行（无容错解析器）时，5 章中 2 章失败。根因：**glm-5.2 在中文摘要里用未转义的 ASCII 双引号包裹对话**，如 `summary: "...只说"再说吧"，随后..."`，破坏 JSON 结构（position 169 是 ASCII `"` 而非全角）。

这是中文 LLM 输出 JSON 的经典坑。已实现 `parseJSONRobust`（`src/json-util.ts`）：逐级尝试 → 直接parse / 去markdown / 提取对象 / **修复未转义引号**。该容错解析器一次性修复了所有失败用例，重试机制甚至未被触发。

**对设计文档的影响**：设计文档修订12（JSON Schema 校验+重试）方向正确，但**需要补充"容错解析"作为校验前的第一步**。仅靠重试效率低（费 token），容错解析能就地修复大多数格式问题。

## 三、验证点②：R2 五维评分是否截断 ✅

**设置**：5 章 Map 结果作为输入，单次调用，`max_tokens: 8192`，要求每维度 100-300 字 analysis + ≥1 条 quote。

**结果**：1 次成功，输出 1986 token，五维完整，无截断。

| 维度 | 分数 | 子项 |
|------|------|------|
| 故事架构 storyStructure | 72 | 情节完整性70 / 叙事节奏75 / 悬念设置68 |
| 人物塑造 characterization | 78 | 立体感80 / 弧光75 / 动机78 / 关系80 |
| 文笔质量 writingQuality | 76 | 表现力78 / 对话80 / 修辞72 / 风格78 |
| 情感共鸣 emotionalResonance | 82 | 张力80 / 代入85 / 节奏78 / 共鸣85 |
| 市场潜力 marketPotential | 65 | 定位70 / 受众68 / 创新60 / 差异化62 |

**质量评估（优秀）**：
- 评分有合理的长短板差异（65-82，情感共鸣最强、市场潜力最弱），未出现"一律给高分"
- 每条 analysis（200-400字）引用了具体章节和情节（"第五章雨夜陈默经过林晚家楼下注意到三楼的灯亮着"）
- 评分标尺锚点生效：文笔因"克制与留白"得 76，市场因"归乡开书吧设定同质化"得 65——判断都有依据
- **零 AI 味套话**（无"综上所述""值得点赞"等）

**对设计文档的影响**：设计文档修订3（Reduce 拆为 R1-R4 防截断）**方向正确，但 spike 显示 R2 单独跑时 1986 token 远未逼近 8192 上限**。对于 ≤200 章的常规小说，R2 单次调用足够；树形 Reduce（>200章）仍需保留作为极端情况兜底。**结论：R2 拆分验证通过，可放心采用单次调用。**

## 四、验证点③：quotes 回链命中率 🔴（重大发现）

**结果：0% 命中（0/10 精确，0/10 模糊，10/10 未命中）。**

**根因分析**：

这是设计文档的一个**根本性缺陷**，spike 把它暴露了出来。

R2 的输入是 Map 阶段的**摘要**（设计文档第219行：`传入全部摘要`），LLM 在 R2 时**根本看不到章节原文**。尽管 prompt 明确要求"逐字摘录原文，一字不改"，glm-5.2 **无法做到**——它只能根据摘要"复述"或"改写"。

实测对比：
- 原文（ch002）：`她想说："你走吧，别回头。"他真的没有回头。`
- R2 quote 输出：`陈默回忆十二年前离开时林晚送他并说'别回头'` ← **这是改写，不是摘录**

10 条 quote 全部是这种"基于摘要的改写概述"，没有一条是原文逐字片段，所以回链精确匹配和模糊匹配全部失败。

**这不是 prompt 工程问题，是架构问题。** R2（基于摘要的全局评判）与"逐字引用原文"在信息层面就是矛盾的——摘要丢了原文措辞。

### 对设计文档的影响（需修订 quotes 机制）

设计文档 v2.1 的 quotes 设计基于一个隐含假设："R2 能逐字引用原文"。这个假设不成立。可行的修订方案有三种，按推荐度排序：

**方案 A（推荐）：把 quotes 生成下放到 Map 阶段。**
- Map 在产出摘要/张力/事件/人物时，**同时为每个维度摘录该章的代表性原文片段**（Map 看得到原文，能逐字摘录）。
- R2 不再产 quotes，而是引用 Map 已摘录的片段。R2 的 analysis 引用 `chapterId`，前端从 Map 结果里取该章的原文片段展示。
- 回链也在 Map 阶段做（原文在手，命中率高）。
- 代价：Map 的输出从 4 项增加到 5 项，prompt 变复杂，每章 token 略增。

**方案 B：R2 产"引用指针"，后端回原文取片段。**
- R2 输出 `{chapterId, searchAnchor: "描述性定位"}` 而非逐字文本。
- 后端根据 searchAnchor 在该章原文里检索最匹配的段落（可用嵌入相似度或关键词）。
- 代价：引入向量检索，复杂度高，命中精度不确定。

**方案 C（最简，降级）：放弃精确逐字引用，接受"概述性引用"。**
- 明确 quotes 就是 LLM 基于摘要的复述，不做 offset 回链。
- 报告里 quotes 只展示为"分析依据"，不支持点击跳转原文。
- 代价：失去"跳转源文件"这个核心交互（设计文档第147行），产品体验降级。

**spike 建议**：采用**方案 A**。它最符合设计初衷（保留跳转原文能力），且 Map 阶段天然能逐字摘录。需修订设计文档的 Map/R2 schema 与 prompt 设计章节。

## 五、验证点④：成本 ✅（超预期）

**实测（5 章，约 2850 字）**：
- Map：input 5270 + output 1143 = **0.0641 元**
- R2：input 2208 + output 1986 = **0.0419 元**
- 合计：**0.106 元**

**外推到 20 万字小说（按 2500 字/章 ≈ 80 章）**：
- Map：input 84320 + output 18288 ≈ **1.03 元**
- R2：input 23328（摘要总量增长）+ output 1986 ≈ **0.25 元**
- 合计：**约 1.3 元/本**

**对比设计文档预估**：设计文档 v2.1 基于 Anthropic 预估"优化后 $5-10/本 ≈ 35-70 元"。实测 GLM 方案 **1.3 元/本，低 30-50 倍**。

> 注：spike 样本每章仅 570 字（偏短），真实网文每章 2500 字会使 Map 的 input token 增大约 4 倍。但即便按此修正，20 万字全量评估也在 **3-5 元** 量级，仍远低于 Anthropic 方案。若采用方案 A（Map 多产 quotes），每章 input 不变、output 略增（+200 token），对总量影响 <20%。

**对设计文档的影响**：成本数据强支撑"用 GLM 替代 Anthropic 作为默认引擎"。设计文档 v2.1 整个引擎层（`AnthropicAPIAdapter` 默认、`maxBudgetUsd`、`cache_control`）需要修订为 GLM 优先。

## 六、附：智谱 Anthropic 兼容端点支持度

spike 顺带验证了 `https://open.bigmodel.cn/api/anthropic` 对 Anthropic 原生字段的支持：

| 字段 | 支持情况 | 证据 |
|------|---------|------|
| `/v1/messages` 基础调用 | ✅ 完全支持 | 所有调用 200，响应格式标准 |
| `system` 参数 | ✅ 支持 | string 形式 |
| `output_schema`（结构化输出）| ⚠️ **字段被接受但不强制** | 调用未报错，但 LLM 仍输出未转义引号的非严格 JSON。兼容层未做 schema 约束，**必须靠后端 parseJSONRobust + 校验** |
| `cache_control`（prompt caching）| ⚠️ 计量字段存在，实际命中未验证 | usage 里有 `cache_read_input_tokens` 字段（ping 测试时为0）；spike 未做 A/B 对比，留待正式开发验证 |
| `temperature` / `max_tokens` | ✅ 支持 | |
| `usage` token 计量 | ✅ 正常返回 | input_tokens / output_tokens 准确 |
| 认证（`x-api-key` + `authorization: Bearer`）| ✅ 任一可用 | |

**关键结论**：兼容端点可用，但 `output_schema` **不可依赖**——它不强制约束输出，仍需后端容错解析+校验+重试三重保险。这点必须在设计文档里写明。

## 七、Go/No-Go 决策

### 可以开工的部分 ✅
- 整体 Map-Reduce pipeline 架构成立
- Map 阶段的摘要/张力/事件/人物提取质量高、稳定
- R2 五维评分质量高、不截断、评分有区分度
- 成本极具优势（1.3 元/本）
- JSON 容错解析方案有效

### 必须先修订再开工的部分 🔴
1. **quotes 机制重构**（采用方案 A：quotes 下放到 Map）——这影响 Map/R2 的 schema、prompt、回链逻辑，是开工前的阻塞项
2. **设计文档引擎层改 GLM 优先**——`AnthropicAPIAdapter` 默认改为 GLM 兼容端点调用，`maxBudgetUsd` 改人民币计价，删 `cache_control` 相关代码（改依赖兼容层隐式缓存）

### 建议但非阻塞 ⚠️
3. spike 样本是合成的（非真实长篇），评分质量验证有局限——建议正式开发时用 golden case（真实小说）回归
4. prompt 全文已落库（`config/prompts/map.md`、`reduce-r2.md`），但仅基于 5 章调优，需在更多样本上迭代
